# 🛠️ 《晨读刮刮乐》前端技术架构与实现指导

## 一、 核心技术栈与工程化配置

为了兼顾开发效率、代码质量和极致的UI定制化需求，项目采用以下技术栈：

*   **核心框架:** **Vue 3** (严格使用 `<script setup>` 组合式 API)
*   **开发语言:** **TypeScript** (开启 strict 模式，杜绝 `any`)
*   **构建工具:** **Vite** (极速冷启动与 HMR)
*   **CSS 框架:** **Tailwind CSS** (原子化 CSS，便于快速构建响应式和定制化拟态风格)
*   **状态管理:** **Pinia** (替代 Vuex，轻量且与 TS 结合完美)
*   **通用 Hooks 库:** **VueUse** (`@vueuse/core`，强烈建议使用其 `useLocalStorage`、`useMediaControls` 等能力)

### 📦 组件库推荐：shadcn-vue
结合 PRD 中要求的“拟态化、可爱化、高度定制UI”，传统的大型组件库（如 ElementPlus/AntD）由于自带强烈的企业级UI风格，重写样式成本极高。
**因此，架构组决定采用 [shadcn-vue](https://www.shadcn-vue.com/)。**
*   **原因:** 它不是一个通过 npm 安装的包，而是直接将组件源码复制到你的项目中（借助 Tailwind）。你拥有 100% 的样式控制权，可以极其方便地通过修改 Tailwind classes 来实现 PRD 要求的圆角、阴影、微渐变等拟态可爱风格。
*   **主要使用组件:** `Dialog`/`Sheet` (用于控制中心弹窗)、`Slider` (音量阈值滑块)、`Button`、`Input`。

### 🎨 第三方特效依赖 (按需引入)
严格遵照 PRD 要求，使用以下开源库：
*   **满屏彩带特效:** `canvas-confetti` (轻量级，性能好)
*   **刮刮卡特效:** `vue3-scratchcard` (注意：PRD 中要求进度满后“自动刮开”，后续会说明实现策略)

## 二、 核心类型定义 (TypeScript Interfaces)

在 `src/types/index.ts` 中定义全局类型，约束数据结构：

```typescript
// 奖励项结构
export interface RewardItem {
  id: string; // 唯一标识 (UUID)
  name: string; // 奖励文案
  weight: number; // 抽中权重 (整数)
}

// 游戏配置结构 (需持久化)
export interface GameConfig {
  mode: 'slot' | 'scratch'; // 当前模式：刮图片(老虎机) / 刮刮乐
  targetDb: number; // 达标音量阈值 (0-120)
  targetDurationSeconds: number; // 目标朗读时长(秒)
  rewards: RewardItem[]; // 奖池列表
}

// 运行时状态结构 (不持久化)
export interface GameState {
  currentDb: number; // 当前实时麦克风音量
  progress: number; // 当前进度 (0-100)
  status: 'idle' | 'recording' | 'finished' | 'celebrating'; // 状态机
  winningReward: RewardItem | null; // 本轮计算出的最终奖品
}
```

## 三、 数据状态管理与持久化 (Store 架构)

在 `src/stores/useGameStore.ts` 中使用 Pinia 建立全局数据中心。**持久化工作交由 VueUse 的 `useStorage` 实现，优雅且安全。**

```typescript
import { defineStore } from 'pinia'
import { ref } from 'vue'
import { useStorage } from '@vueuse/core'
import type { GameConfig, GameState, RewardItem } from '@/types'

export const useGameStore = defineStore('game', () => {
  // 1. 持久化配置状态 (自动同步 LocalStorage)
  const config = useStorage<GameConfig>('chen-du-config', {
    mode: 'scratch',
    targetDb: 60,
    targetDurationSeconds: 60, // 默认 1 分钟
    rewards: [
      { id: '1', name: '零食一份', weight: 5 },
      // ... 注入 PRD 中所有的默认内置奖励
    ]
  })

  // 2. 运行时挥发状态 (每次刷新重置)
  const runtimeState = ref<GameState>({
    currentDb: 0,
    progress: 0,
    status: 'idle',
    winningReward: null
  })

  // 3. 核心 Action：权重随机算法 (需在达到 100% 之前调用计算好)
  const calculateWinner = () => {
    const pool = config.value.rewards;
    if (!pool.length) return;
    
    const totalWeight = pool.reduce((sum, item) => sum + item.weight, 0);
    let randomNum = Math.random() * totalWeight;
    
    for (const item of pool) {
      if (randomNum < item.weight) {
        runtimeState.value.winningReward = item;
        return;
      }
      randomNum -= item.weight;
    }
  }

  // 4. 重置游戏
  const resetGame = () => {
    runtimeState.value.progress = 0;
    runtimeState.value.status = 'idle';
    runtimeState.value.winningReward = null;
  }

  return { config, runtimeState, calculateWinner, resetGame }
})
```

## 四、 核心底层能力实现 (Composables Hooks)

这是本项目最硬核的部分，必须实现逻辑与 UI 的完全解耦。

### 1. 音频处理引擎 `useAudioAnalyzer.ts`
负责向浏览器索要权限，利用 `AudioContext` 实时分析麦克风输入的频率数据，转化为可视化所需的 DB 值。

*实现要点:*
*   处理浏览器限制（必须由用户点击按钮触发后才能创建 `AudioContext`）。
*   使用 `AnalyserNode` 获取频域数据，并计算均方根 (RMS) 以获得较为平滑的音量值，然后映射为 0-120 分贝的相对值。
*   **极其重要:** 在 `onUnmounted` 时，必须调用 `audioContext.close()` 和 `cancelAnimationFrame` 释放内存，否则会导致浏览器严重卡顿。

### 2. 游戏进度引擎 `useGameEngine.ts`
基于 `useAudioAnalyzer` 抛出的实时 DB 值，与 Store 中的配置进行比较，驱动进度条。

*实现要点:*
*   监听 DB 值：`if (currentDb >= config.targetDb) { /* 累加进度 */ } else { /* 暂停累加 */ }`
*   使用 `requestAnimationFrame` 来累加时间戳 `deltaTime`，而不是使用 `setInterval`，以保证进度条推进的丝滑度和计时的准确性。
*   当进度 `>= 100` 时：
    1.  停止音频监听。
    2.  调用 `store.calculateWinner()` 算出奖品。
    3.  更改状态 `store.runtimeState.status = 'finished'`，通知 UI 层触发特效。

### 关键 UI 组件技术指导：

#### 1. 进度条 (ProgressBar.vue)
*   请使用 Tailwind 的 `transition-all duration-300 ease-out` 配合动态绑定 `:style="{ width: progress + '%' }"` 实现丝滑推进。
*   不达标时，给进度条容器添加一层灰色的 `filter: grayscale(100%)` 或 CSS 呼吸闪烁动效 (pulse)。

#### 2. 刮图片模式 (RewardSlotMode.vue)
*   在 `status === 'recording'` 且分贝达标时，使用 `setInterval` 每 100ms 随机切换当前显示的奖励文案（造出盲盒感）。
*   在 `status === 'finished'` 时，停止定时器，直接展示 Store 中计算好的 `winningReward.name`，并加上一个 CSS 放大的弹簧动画 (Bounce)。

#### 3. 刮刮乐模式 (RewardScratchMode.vue) 与 vue3-scratchcard 整合
**架构师特别提醒：** PRD 提出的是“**进度满了自动刮开**”，但标准的刮刮卡组件通常是**用户手动用鼠标刮开**。
*   **开发方案设定：** 引入 `vue3-scratchcard`。
    *   底层放置已经渲染好 `winningReward.name` 的 DOM。
    *   表层覆盖刮刮卡组件（加载彩色的覆盖层图片）。
    *   **实现“自动刮开”：** 当 `status === 'finished'` 达到 100% 时，利用 Vue 的 `<Transition>`，直接让表层的 `vue3-scratchcard` 容器产生一个 `opacity: 0` 或者 `clip-path` 的 CSS 消失/擦除动画。这样既利用了组件生成遮罩层，又完美契合了 PRD“自动呈现”的视觉要求，避免了修改第三方组件源码的麻烦。

#### 4. 满屏彩带与高光弹窗 (ConfettiModal.vue)
*   利用 Vue 的 `<Teleport to="body">` 保证弹窗层级最高。
*   使用 PRD 指定的 `canvas-confetti` 库。
*   **调用时机:** 在弹窗组件 `onMounted` 或监听到 `visible = true` 时，执行 `confetti({ particleCount: 150, spread: 80, origin: { y: 0.6 } })`。可以组合调用两次营造更盛大的效果。

## 六、 性能优化与兼容性底线 (Checklist)

一线研发在开发过程中，必须时刻校验以下事项：

1.  **硬件加速 (GPU Acceleration):** 进度条填充、弹窗缩放动效必须使用 `transform` 和 `opacity` 实现，严禁使用 `width/height/margin` 来做动画，保证在学校老旧的多媒体大屏上不掉帧。
2.  **避免频繁的响应式依赖触发:** `useAudioAnalyzer` 产生的 `currentDb` 是高频变化的数据（一秒几十次），不要直接把它绑定到极其复杂的 DOM 树上，只绑定到对应的**数字展示**和**进度条小组件**内部，防止全局 Vue 虚拟 DOM 的重绘。
3.  **UI 响应式断点:**
    *   大屏 (多媒体教室): `xl:text-5xl`, `xl:p-12` (字体要足够大)。
    *   平板 (教师巡查): 重点保证 Control Panel 在横屏和竖屏下都能滚动显示完所有设置项。
